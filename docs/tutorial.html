> agentic-loop

Multi-model orchestration for Claude Code. Plan, delegate, review, ship.

Level 0

What is this and how do I install it?

The agentic-loop plugin turns your Claude Code session into an orchestrated multi-model system. Your interactive Opus session plans and routes work to the cheapest model that can handle it — Haiku for mechanical tasks, Sonnet for judgment calls, Fable for frontier reasoning, and external models (Sol, OpenRouter, Ollama) via bash scripts. A blind review chain catches errors before they ship.

1

Check prerequisites

  • Claude Max subscription — the interactive session and native subagents bill here
  • Claude Code CLI installed and logged in
  • jq and curl on your PATH
  • Optional: API keys for Fable, Sol, OpenRouter (for metered tiers); Ollama running locally (for free tier)
2

Install the plugin

Clone (or use an existing checkout), register, then install:

git clone https://github.com/thedull/agentic-loop /path/to/agentic-loop
claude plugin marketplace add /path/to/agentic-loop
claude plugin install agentic-loop

The path can be anywhere on disk — if you already have a checkout, skip the clone step.

Or load it for a one-off session without installing:

claude --plugin-dir /path/to/agentic-loop
3

Scaffold your project

Inside any project directory, run the init skill:

/agentic-loop:init

This copies the worker scripts, routing-policy CLAUDE.md, .env.example, and the .agentic/ run-state scaffold into your project.

4

Configure your keys

cp .env.example .env
# Edit .env — fill in the keys you have

You only need the keys for tiers you plan to use. Skip any you don't have — the corresponding script fails cleanly with a structured error.

Billing trap

Never set ANTHROPIC_API_KEY in your shell environment or .env. If it's set, Claude Code silently switches your interactive session from Max-subscription billing to metered API billing. The Fable worker key is deliberately named FABLE_KEY to avoid this.

5

Run doctor

./scripts/doctor.sh

Doctor checks for the billing trap, verifies your tools are installed, validates your keys are in .env (not exported), and runs the envelope validator's self-test. Fix any FAIL lines before proceeding.

All checks passing? You're ready. Move on to Level 1 for your first loop.

Level 1

Run your first loop

Give the orchestrator a small task and watch the full loop in action. Here's a concrete example you can run in any codebase: 

Summarize the 3 largest files in this repo. Use the agentic loop:
plan the work, delegate to workers, consolidate, and review.

The flow

The orchestrator breaks this into a pipeline:

@loop-planner → @loop-worker-cheap → @loop-consolidator → @loop-reviewer
P

Plan

@loop-planner (Sonnet) decomposes the task into subtasks, each with a 6-field delegation brief: objective, user intent verbatim, input paths, boundaries, output spec, and effort budget. It assigns each subtask to the cheapest adequate tier and marks which can run in parallel.

D

Delegate

@loop-worker-cheap (Haiku) executes each brief mechanically. For this example: find the 3 largest files, read them, produce summaries. Each worker returns a JSON envelope with its results.

C

Consolidate

@loop-consolidator (Sonnet) merges the worker outputs. It verifies artifacts exist on disk, deduplicates, and surfaces any disagreements between workers without forcing consensus. The merged result goes to .agentic/artifacts/.

R

Review

@loop-reviewer (Sonnet) does a blind review: it sees the task spec and the artifact, but not the reasoning that produced it. It runs any executable checks it can and reports only correctness gaps with cited evidence.

Reading the envelope

Every worker returns the same JSON structure. The fields that matter most:

status
ok = done, partial = incomplete, error = failed, needs_input = questions for you
summary
100-word digest of what was done
result
The actual output, per the brief's output spec
artifacts[]
Paths to files the worker wrote (orchestrator verifies they exist)
caveats[]
Anything that limits trust in this result

When things go sideways

  • needs_input — the worker has questions. Answer them and re-run.
  • error — check the summary field for what went wrong. Often a missing file or a command that failed.
  • needs_escalation — the worker or reviewer flagged this for a higher tier. The orchestrator decides whether to escalate.
Full envelope schema 
{
  "worker": "haiku",
  "status": "ok|partial|error|blocked|needs_escalation|needs_input",
  "summary": "<=100 words",
  "result": { },
  "artifacts": ["paths to written files"],
  "key_decisions": ["choices the worker made"],
  "caveats": ["trust limiters"],
  "assumptions": ["things the brief left unspecified"],
  "confidence_ordinal": "high|medium|low",
  "usage": {
    "input_tokens": 0,
    "output_tokens": 0,
    "est_cost_usd": 0
  }
}

Level 2

Use every tier and optimize cost

The tier ladder

Route by difficulty and stakes, not by default. Start cheap; escalate on measured failure.

FREE

ollama

Local model via call_ollama.sh. Mechanical: extraction, formatting, bulk lookups. Default: qwen3.5:4b.

SUB

haiku

@loop-worker-cheap subagent. Mechanical with repo access and tool use.

SUB

sonnet

@loop-planner, @loop-consolidator, @loop-reviewer. Judgment: decomposition, merging, review.

SUB

fable (native)

@loop-frontier, @loop-reviewer-frontier. Subscription-covered while your plan includes Fable. Pre-Sol review tier.

$

openrouter

call_openrouter.sh --model kimi|minimax|mimo. Cheap bulk generation and second opinions.

$$

fable (script)

call_fable.sh. Same capability as native, metered via FABLE_KEY. Use when Fable leaves your subscription plan.

$$$

sol

call_sol.sh. Cross-family adversary. Structural triggers only — the human is the circuit breaker.

Smoke-test each tier

Quick one-shot tests to verify each tier works. Native subagents are subscription-covered; script calls are metered (pennies each).

# Native subagents (subscription)
@loop-worker-cheap "List all .sh files in scripts/ and count them"
@loop-frontier "Explain what scripts/common.sh does in 3 bullets"

# Bash shim workers (metered — needs keys in .env)
./scripts/call_ollama.sh --objective "Reply with the word ok"
./scripts/call_fable.sh --objective "Reply with the word ok"
./scripts/call_sol.sh --mode reviser --effort standard \
  --objective "Reply with the word ok"
./scripts/call_openrouter.sh --model kimi \
  --objective "Reply with the word ok"

The review chain

For high-stakes output, the review escalates through tiers:

@loop-reviewer → @loop-reviewer-frontier → call_sol.sh --mode adversary

Each reviewer is blind — it sees the task spec and the candidate artifact, never the reasoning that produced it. The frontier reviewer is same-family (Anthropic), so it provides fresh context and frontier capability but not cross-family independence. Sol (GPT-5.6) is the only cross-family adversary.

Blind adversary protocol

  • Blind payload: task + candidate only. Author's reasoning is withheld.
  • Proof before preference: every finding needs cited evidence (file:line, failing command, spec contradiction) before any verdict.
  • Correctness only: no style, no taste, no hypothetical improvements.
  • Empty findings = valid answer: don't manufacture findings to appear thorough.

Structural escalation triggers

Sol is expensive. It fires only on structural triggers, never on self-reported confidence:

  1. High-stakes or irreversible output (production deploy, data migration)
  2. Material disagreement between worker and reviewer that persists after revision
  3. Known-hard task class (security, concurrency, distributed state)
  4. Repeated worker or schema failure (something is systematically wrong)
  5. Tests failing after a revision attempt

Cheap iteration mode

Run early iterations with /model claude-sonnet-4-6 as your orchestrator. This burns the Sonnet-specific weekly cap instead of the all-models cap, preserving Opus and Fable quota for the hard passes. Two rules:

  • Follow the structural escalation triggers mechanically — don't substitute your confidence for them
  • Switch /model up to Opus or Fable for final consolidation and the ship decision

Level 3

Headless, hardening, and customization

Headless runs

run_headless.sh wraps claude -p for unattended loops. It bills on a different meter than your interactive session and refuses to run without explicit acknowledgment: 

./scripts/run_headless.sh \
  --i-understand-billing \
  --prompt-file task.md \
  --check-cmd "test -f output.json" \
  --max-iterations 5
  • --i-understand-billing — required gate, no shortcut
  • --prompt-file — must exist, contains the task
  • --check-cmd — completion check, runs after each iteration
  • --max-iterations — hard cap (default 5), exits code 6 on exhaustion
  • --dry-run — shows what would run without executing

Spawn-guard hook

Caps concurrent subagent spawns to prevent runaway fan-out. Merge the template into your project's settings: 

# From templates/hooks-spawn-guard.json
# Merge into .claude/settings.json
# MAX_PARALLEL=6 within a 60-second window

The .agentic/ blackboard

The .agentic/ directory is the shared coordination layer during a run. It's gitignored — it's ephemeral state, not project code.

  • PLAN.md — orchestrator-only. The current decomposition and routing. Survives context compaction.
  • STATUS.md — orchestrator-only. Milestone log: what's done, what's next.
  • decisions.md — append-only, all workers read. Records choices so no one re-litigates.
  • artifacts/ — single-writer-per-file. Workers write here; the orchestrator and consolidator read.

Adding new worker scripts

Follow the shim pattern in scripts/lib/common.sh:

  1. Source common.sh for .env loading, brief parsing, and envelope building
  2. Use require_key to check for your API key
  3. Use parse_brief to read the 6-field delegation brief
  4. Use build_task_prompt + envelope_instructions to construct the prompt
  5. Call your model's API with curl
  6. Use finalize_envelope to parse, validate, and return
Minimal script skeleton 
#!/usr/bin/env bash
set -euo pipefail
SCRIPT_DIR="$(cd "$(dirname "$0")" && pwd)"
source "$SCRIPT_DIR/lib/common.sh"

require_key "MY_API_KEY"

parse_brief "$@"
PROMPT="$(build_task_prompt)"
SYS="You are a worker.\n$(envelope_instructions)"

RESPONSE=$(curl -s "https://api.example.com/v1/chat" \
  -H "Authorization: Bearer $MY_API_KEY" \
  -H "Content-Type: application/json" \
  -d "$(jq -n --arg p "$PROMPT" --arg s "$SYS" \
    '{model:"example-1", messages:[
      {role:"system",content:$s},
      {role:"user",content:$p}
    ]}')")

OUTPUT=$(echo "$RESPONSE" | jq -r '.choices[0].message.content')
finalize_envelope "$OUTPUT" --worker "example"

LEARNINGS.md and the two-strikes rule

LEARNINGS.md is the cross-run memory file. Workers and reviewers record patterns on the second occurrence — one-offs aren't patterns. Cap at ~300 lines; prune entries that no longer apply.

Why Sol can't be replaced

The frontier-native reviewers (@loop-reviewer-frontier) are same-family: they share training data, failure modes, and blind spots with the author models. Sol (GPT-5.6) is the only cross-family adversary in the loop — its independent training catches failure modes that Anthropic models are structurally unlikely to find in each other's work. This is the one tier where "different vendor" is the feature.
